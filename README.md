## Многопоточная сортировка слиянием и моделирование планировщиков процессов
### Общее описание проекта

Данный проект представляет собой комплексную реализацию двух ключевых алгоритмов операционных систем: 
многопоточной сортировки слиянием (parallel merge sort) и моделирования планировщиков процессов (process schedulers). 
Проект демонстрирует практическое применение примитивов синхронизации, 
работу с потоками и реализацию алгоритмов вытеснения страници планирования CPU.

Проект написан на языке C с поддержкой кросс-платформенности (Windows и Linux). 
Вся логика покрыта автоматическими тестами, проверяющими корректность работы алгоритмов, синхронизацию потоков и производительность.

### Часть 1: Многопоточная сортировка слиянием
#### Классическая сортировка слиянием
Сортировка слиянием (Merge Sort) - это алгоритм сортировки, основанный на принципе "разделяй и властвуй". 
Алгоритм рекурсивно делит массив на две половины, сортирует каждую половину, а затем сливает отсортированные половины в один массив.

Сложность алгоритма:
- Временная сложность: O(n log n) во всех случаях
- Пространственная сложность: O(n) для временных массивов
- Устойчивая сортировка (сохраняет порядок равных элементов)

#### Параллельная сортировка слиянием
В классической реализации сортировка слиянием выполняется последовательно, что не использует преимущества многоядерных процессоров. 
Параллельная версия алгоритма позволяет одновременно выполнять сортировку разных частей массива.

Ключевые принципы параллелизации:
1. Параллельная сортировка половин: Левая и правая половины массива могут сортироваться одновременно в разных потоках
2. Параллельное слияние с бинарным поиском. Вместо последовательного слияния,
   алгоритм использует бинарный поиск для нахождения позиций элементов, что позволяет параллельно сливать части массивов.
3. Порог последовательной сортировки. Для небольших массивов (менее 5000 элементов) используется последовательная сортировка,
   чтобы избежать накладных расходов на создание потоков.

#### Алгоритм параллельного слияния с бинарным поиском
Этот алгоритм является ключевым для достижения параллелизма в сортировке слиянием.

Шаги алгоритма:
1. Проверка базовых условий
2. Балансировка массивов
3. Выбор опорного элемента
4. Бинарный поиск во втором массиве
5. Вставка опорного элемента
6. Рекурсивное параллельное слияние

#### Управление потоками и синхронизация
##### Счетчик активных потоков
Для контроля количества создаваемых потоков используется глобальный счетчик active_threads. 
Каждый раз перед созданием нового потока проверяется, не превышен ли лимит.

##### Проблема гонки данных. 
Несколько потоков могут одновременно пытаться изменить счетчик, что приведет к некорректным значениям.

Решение: Использование критической секции (CRITICAL_SECTION) для атомарного обновления счетчика.

#### Синхронизация завершения потоков
Для ожидания завершения созданных потоков используется функция WaitForSingleObject (Windows) или pthread_join (Linux). 
Это гарантирует, что все потоки завершат работу до того, как будет использован результат.

### Пороги оптимизации
В реализации используются пороги для баланса между параллелизацией и накладными расходами:

- Порог сортировки (5000 элементов): Если размер подмассива меньше этого значения, используется последовательная сортировка.
- Порог глубины рекурсии (4): Ограничивает глубину параллельной рекурсии, предотвращая создание слишком большого количества потоков.
- Порог слияния (2000 элементов): Если общий размер двух массивов меньше этого значения, используется последовательное слияние.

Преимущества:
- Эффективное использование многоядерных процессоров
- Сложность O(log² n) при параллельном выполнении
- Масштабируемость с ростом числа ядер

Недостатки:
- Накладные расходы на создание потоков
- Сложность синхронизации
- Снижение эффективности на маленьких массивах (что наблюдается в тестах)

### Часть 2: Моделирование планировщиков процессов
Планирование процессов (CPU scheduling) - это механизм операционной системы, который определяет, 
какой процесс получит доступ к процессору и на какое время. 
Основная задача планировщика - эффективно распределять ресурсы CPU между конкурирующими процессами.

Критерии оценки планировщиков
- Время ожидания (Waiting Time): Суммарное время, которое процесс провел в очереди готовых процессов.
- Время оборота (Turnaround Time): Время от момента поступления процесса в систему до его завершения.
- Время отклика (Response Time): Время от поступления процесса до первого получения CPU.
- Пропускная способность (Throughput): Количество процессов, завершаемых в единицу времени.
- Загрузка CPU (CPU Utilization): Процент времени, когда процессор занят полезной работой.


### Реализованные алгоритмы планирования
#### 1. FIFO (First In, First Out) - невытесняющий
Принцип работы:
- Процессы выполняются в порядке их поступления в систему
- Первый прибывший процесс выполняется первым
- После запуска процесса он выполняется до полного завершения
- Невытесняющий алгоритм - процесс не может быть прерван

#### 2. Round Robin (RR) - вытесняющий
Принцип работы:
- Каждому процессу выделяется квант времени (обычно 10-100 мс)
- Процесс выполняется в течение кванта, затем помещается в конец очереди
- Если процесс завершается до истечения кванта - он удаляется из очереди
- Вытесняющий алгоритм - процесс может быть прерван по истечении кванта

#### 3. Приоритетное планирование (Priority) - вытесняющий
Принцип работы:
- Каждому процессу назначается приоритет (число)
- Процесс с наивысшим приоритетом (наименьшее число) выполняется первым
- Вытесняющий вариант: если появляется процесс с более высоким приоритетом, текущий процесс прерывается
- Невытесняющий вариант: процесс выполняется до завершения, даже если появляется более приоритетный

### Вытеснение процессов
Вытеснение (Preemption) - это механизм, при котором текущий процесс может быть принудительно остановлен, чтобы передать CPU другому процессу.

#### Условия для вытеснения:
- Истечение кванта времени (RR): Процесс выполнялся заданное время
- Появление более приоритетного процесса (Priority): Вновь прибывший процесс имеет более высокий приоритет
- Завершение текущего процесса: Естественное завершение работы

### Часть 3: Синхронизация потоков
#### Теория синхронизации
В многопоточных приложениях несколько потоков могут одновременно обращаться к общим данным, 
что может привести к состоянию гонки (race condition). 
Для предотвращения этого используются примитивы синхронизации:

1. Критические секции (Critical Sections)
Критическая секция - это участок кода, который должен выполняться атомарно, без прерываний от других потоков.

Принцип работы:
- Поток входит в критическую секцию (захватывает блокировку)
- Выполняет защищенный код
- Выходит из критической секции (освобождает блокировку)

2. Ожидание завершения потоков
Для синхронизации завершения потоков используется механизм ожидания.

3. Защита счетчика активных потоков
Проблема: Несколько потоков одновременно пытаются увеличить счетчик, атомарно.

#### Возможные блокировки (Deadlocks)
Deadlock (взаимная блокировка) - ситуация, когда два или более потоков ожидают ресурсы, занятые друг другом, 
и ни один из них не может продолжить выполнение.

Условия возникновения deadlock:
- Взаимное исключение: Ресурс может быть занят только одним потоком
- Удержание и ожидание: Поток удерживает ресурс и ждет другой
- Отсутствие вытеснения: Ресурс не может быть принудительно отобран
- Циклическое ожидание: Цепочка потоков, ожидающих ресурсы друг друга

В проекте deadlock предотвращается:
- Использование одного мьютекса для защиты всех общих данных
- Отсутствие вложенных блокировок
- Короткое время удержания мьютекса

### Часть 4: Тестирование
#### Обзор тестов
Проект включает 12 автоматических тестов, покрывающих все аспекты функциональности:

1. Последовательная сортировка - проверка корректности базового алгоритма
2. Сравнение сортировок - оценка ускорения параллельной версии (в данном случае, из-за больших затрат на создание потоков, ускорение уменьшается)
3. FIFO планировщик - проверка невытесняющего алгоритма
4. Round Robin планировщик - проверка вытесняющего алгоритма с квантованием
5. Priority планировщик - проверка приоритетного вытесняющего алгоритма
6. Сравнение планировщиков - оценка эффективности всех трех алгоритмов
7. Проверка синхронизации - тестирование примитивов синхронизации
8. Стресс-тест сортировки - проверка на разных размерах данных
9. Отсортированный массив - проверка на уже отсортированных данных
10. Все равные элементы - проверка на массиве с одинаковыми значениями
11. Обратный порядок - проверка на массиве по убыванию
12. Различное число потоков - проверка масштабируемости
